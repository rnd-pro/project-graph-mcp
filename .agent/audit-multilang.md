# Аудит мультиязычного парсера project-graph-mcp

## 1. `src/lang-typescript.js` (НОВЫЙ)
*   **[OK]** Все ли TS конструкции корректно стрипятся: Да, типы, интерфейсы, модификаторы доступа и дженерики успешно удаляются.
*   **[OK]** Сохраняется ли длина строк (замена на пробелы а не удаление): Да, используется функция `space()`, которая заменяет символы на пробелы (`[^\r\n]`), сохраняя позиции в AST.
*   **[ISSUE]** Нет ли проблем с template literals, JSX в TSX: Есть проблемы. Поскольку удаление работает через регулярные выражения (например, `\btype\b`), парсер будет ошибочно удалять/заменять эти ключевые слова внутри строковых литералов, шаблонных строк (template literals) и JSX-кода.
*   **[OK]** Тестируй edge cases (`export const fn = (a: string): void => {}`): Проверка пройдена, типы удалены с сохранением пробелов.

## 2. `src/lang-python.js` (НОВЫЙ)
*   **[OK]** Корректно ли определяются top-level vs class methods (по отступам): Да, логика сброса скоупа на основе `classIndent` работает корректно.
*   **[OK]** Обрабатываются ли декораторы (`@staticmethod`, `@property`): Да, декораторы просто пропускаются построчным парсером, и метод, идущий следом, распознается верно.
*   **[OK]** Правильно ли работает очистка строк в тройных кавычках: Да, вырезаются до основного парсинга через `replace`, хоть и могут быть редкие ошибки в сложных комбинациях эскейпов.
*   **[ISSUE]** Тестируй: вложенные классы, multiline imports: **Multiline imports не работают.** Тест показал `imports: []`. Парсер читает файл строго построчно, и регулярное выражение `^\s*from\s+([.\w]+)\s+import\s+(.+)` не захватывает импорты, перенесенные на следующие строки. Вложенные классы также могут некорректно перезаписывать `currentClass`.

## 3. `src/lang-go.js` (НОВЫЙ)
*   **[OK]** Корректно ли парсятся methods с pointer receivers (`*Type`) и value receivers (`Type`): Да, регулярное выражение `\*?([a-zA-Z_]\w*)` корректно обрабатывает оба варианта.
*   **[ISSUE]** Обрабатываются ли interface declarations: **Нет**. Регулярное выражение `structRegex` ищет только `struct\s*\{`, поэтому все `interface` игнорируются парсером.
*   **[OK]** Правильно ли извлекается embedded struct (extends): Да, извлекается как первое слово в строке (если нет типа), но захватывается только последний embedded struct, если их несколько.
*   **[OK]** Тестируй: multiline struct с тегами `json:"name"`: Проверка пройдена. Обработка бэктиков (`` ` ``) в `stripStringsAndComments` успешно их удаляет, поэтому теги не ломают извлечение свойств.

## 4. `src/tools.js` (ИЗМЕНЁН)
*   **[OK]** `getCallChain()` — работает ли BFS правильно, нет ли бесконечного цикла: Да, используется множество `visitedNodes` для предотвращения зацикливания.
*   **[OK]** `saveDiskCache`/`loadDiskCache` — обработка ошибок при чтении/записи JSON: Обернуто в `try...catch`, поэтому падения при битых файлах кэша не происходит.
*   **[ISSUE]** Нет ли конфликтов от двух агентов, писавших одновременно: **Возможны конфликты.** `writeFileSync` и `readFileSync` работают без механизма блокировки (file lock), из-за чего при параллельной записи файл кэша может быть поврежден (race condition).
*   **[OK]** Импортируется ли `unlinkSync` если `invalidateCache` удаляет файл: Да, импорт из модуля `fs` присутствует.

## 5. `src/parser.js` (ИЗМЕНЁН)
*   **[OK]** `parseFileByExtension` правильно маршрутизирует: Да, корректный роутинг по расширениям `.ts`, `.py`, `.go`.
*   **[OK]** `isSourceFile` не пропускает нужные файлы: Массив `SOURCE_EXTENSIONS` содержит все необходимые типы.
*   **[OK]** Нет ли дублирования парсинга .ts файлов в `*.d.ts` (они в exclude patterns): Файлы `*.d.ts` не попадают в парсер, так как корректно отфильтровываются через дефолтные `excludePatterns` в модуле `filters.js` (с помощью функции `shouldExcludeFile`).
