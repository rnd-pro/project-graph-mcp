# Исследование: Поддержка нескольких языков программирования в project-graph-mcp

В данном отчете рассмотрены варианты расширения `project-graph-mcp` для поддержки TypeScript, Python, Go, Rust и других языков, с учетом главного архитектурного принципа проекта — **"zero dependencies"** (отсутствие зависимостей в `package.json`). 

В текущей реализации для парсинга JavaScript используется встроенный (vendored) `acorn.mjs`.

---

## 1. tree-sitter через WASM (`web-tree-sitter`)
Использование скомпилированного в WebAssembly движка tree-sitter и WASM-модулей для каждого языка (например, `tree-sitter-typescript.wasm`).

* **Сохраняет ли "zero dependencies":** **Да**, можно скачать `web-tree-sitter.js` и нужные `.wasm` файлы и положить их в папку `vendor/`.
* **Размер:** Движок весит ~100-200 КБ. Парсер каждого языка весит от 100 КБ до 1.5 МБ (TS ~1МБ, Python ~400КБ). Суммарно папка `vendor/` вырастет на 3-5 МБ.
* **Скорость парсинга:** Очень высокая. Движок написан на C/Rust и скомпилирован в WASM, что дает производительность, близкую к нативной (значительно быстрее чистого JS-парсера вроде Acorn).
* **Надежность:** **Очень высокая**. Tree-sitter создан специально для IDE, он строит полное AST, отлично справляется с ошибками синтаксиса и позволяет извлекать вызовы, функции и классы с помощью встроенного механизма запросов (Query).
* **Сложность интеграции:** **Высокая**. Потребуется написать адаптер (и/или tree-sitter queries) для каждого языка, чтобы приводить разные AST к единому формату `ParseResult` (`classes`, `functions`, `imports`, `exports`), который ожидает `graph-builder.js`. Также нужно будет настроить асинхронную инициализацию WASM-модулей.

---

## 2. Отдельные легковесные JS-парсеры
Поиск нативных JS-парсеров для каждого языка (например, TypeScript Compiler API для TS, сторонние проекты для Python/Go).

* **Сохраняет ли "zero dependencies":** **Сложно**. Вендоринг таких парсеров крайне затруднителен.
* **Размер:** Огромный. Только `typescript.js` весит более 10 МБ. Чистых и компактных JS-парсеров для Go и Rust практически не существует (они либо тянут огромный рантайм, либо неполные).
* **Скорость парсинга:** Для TypeScript Compiler API — низкая (он медленный по сравнению с Acorn/tree-sitter). Для других языков — зависит от реализации, но обычно медленно.
* **Надежность:** Отличная для TS (так как это официальный компилятор), но крайне низкая для "самописных" JS-парсеров других языков.
* **Сложность интеграции:** **Очень высокая**. Каждый парсер имеет свою уникальную архитектуру AST. Придется писать сложный слой абстракции (walk/visit) для каждого языка с нуля.

---

## 3. Минимальный regex-based парсинг
Использование регулярных выражений для извлечения сигнатур `class X`, `def func():`, `import Y` без построения полноценного абстрактного синтаксического дерева.

* **Сохраняет ли "zero dependencies":** **Да**, 100% нативный JS (RegExp).
* **Размер:** Минимальный (~10-50 КБ кода в `vendor/` или `src/`).
* **Скорость парсинга:** **Максимальная**. Регулярки работают в разы быстрее любого AST-парсера.
* **Надежность:** **Крайне низкая**. Регулярки ломаются на многострочных строках, комментариях, вложенных структурах (внутренние классы/функции). Невозможно точно отследить цепочки вызовов (calls), что критично для построения графа зависимостей (`graph-builder.js`).
* **Сложность интеграции:** Низкая на старте, но оборачивается огромными затратами на поддержку и обработку краевых случаев (edge cases).

---

## 4. Плагинная архитектура (Language Plugins)
Ядро проекта остается "zero-dependencies" (и парсит только JS через Acorn), но вводится интерфейс для плагинов. Сообщество или пользователь может установить парсер глобально, а `project-graph-mcp` вызовет его.

* **Сохраняет ли "zero dependencies":** **Да** для ядра, но перекладывает зависимости на пользователя.
* **Размер:** Ядро остается компактным. Раздувается только окружение пользователя.
* **Скорость парсинга:** Зависит от плагина.
* **Надежность:** Зависит от реализации плагина.
* **Сложность интеграции:** **Средняя**. Нужно модифицировать `parser.js` для динамической загрузки плагинов (по расширению файла) и стандартизировать API плагина (`Plugin.parse(code) => ParseResult`). Это хороший компромисс, но пользователи не любят "собирать конструктор" при работе с MCP-серверами.

## 5. Вызов `tree-sitter CLI` (через child_process)
Вызов бинарника `tree-sitter parse --output=json <file>` через `child_process.execSync` из Node.js.

* **Сохраняет ли "zero dependencies":** **Да** (на уровне NPM), но создает жесткую системную зависимость — у пользователя должен быть установлен `tree-sitter CLI` в `$PATH`.
* **Размер:** Добавочный вес 0 КБ в репозитории.
* **Скорость парсинга:** **Низкая**. Спавн (spawn) дочернего процесса для каждого файла в крупном проекте (тысячи файлов) создаст огромный overhead. Пакетная обработка затруднена.
* **Надежность:** AST парсится надежно, но могут быть проблемы с путями, кодировками, максимальной длиной командной строки и отсутствием CLI у конечного пользователя.
* **Сложность интеграции:** **Средняя**. Легко прочитать JSON вывод CLI, но сложно обеспечить fallback-механизмы, если CLI не установлен.

## Вывод и рекомендации

Наиболее оптимальным и перспективным вариантом для `project-graph-mcp` является **Вариант 1 (tree-sitter через WASM)**:
1. Он на 100% сохраняет принцип **zero dependencies** в `package.json` (всё можно свендорить).
2. Дает доступ к надежному парсингу (с обработкой ошибок) для десятков языков, включая TypeScript, Python, Go, Rust.
3. Обеспечивает высокую скорость работы.

**План внедрения (Roadmap для WASM):**
1. Добавить в `vendor/` файлы `web-tree-sitter.js`, `tree-sitter.wasm` и `tree-sitter-typescript.wasm`.
2. Создать унифицированный `TreeSitterAdapter`, который инициализирует WASM и выполняет `Query` для извлечения классов, функций и импортов.
3. Подключить `.ts` и `.tsx` файлы в `filters.js`.
4. В `parser.js` добавить роутинг: `.js` парсится через Acorn (чтобы не ломать текущую логику и сохранить макс. скорость для JS), а `.ts`/`.py` направляются в `TreeSitterAdapter`.
